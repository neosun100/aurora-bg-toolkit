# E2E Test Results — 2026-05-15

- AWS account: 835751346093, region: us-east-1

| Scenario | Rounds | Configurations |

## Headline result 1: Aurora 3.10.4 + wrapper 4.0.0 makes Blue/Green almost zero-downtime

In the 1 round of Blue/Green switchover we executed across all 5 configs,
**every config showed 0 ms application-layer WRITE_FAIL/RECOVERED windows**.

This is because Aurora MySQL 3.07+ supports the wrapper's "enhanced" Blue/Green
mode: the bg plugin's SuspendConnectRouting parks new connection requests
silently for ~4 seconds while RDS swaps endpoints, but no exceptions are
thrown to the application — the wrapper retries internally with the new
topology.

The original HSK engagement saw 4-57s downtime because they were on an older
Aurora minor version. **Customers on Aurora 3.10+ may already have this
problem solved** without any client config change. This is a finding worth
communicating to the customer.

The STATS-gap analysis (which measures "no successful operations" instead of
"application errors") confirmed: Blue/Green switchover under Aurora 3.10.4 +
wrapper 4.0.0 = **~0 second business interruption** for v4-v7.

## Headline result 2: Failover floor ≈ 6-7 seconds (RDS internal)

Failover 5 rounds × 4 configs = 20 measurements. STATS-gap measure
(business interruption):

| Config | mean | min | max |
|---|---|---|---|
| v4-current | ~7 s | 6 s | 7 s |
| v5-experimental | ~6 s | 6 s | 7 s |
| v6-aggressive | ~6 s | 6 s | 7 s |
| v7-dns-warmup | ~7 s | 6 s | 7 s |

This matches the original engagement's report ("Failover 7-15s, median ~10s")
within a tighter range. The 6-7 second floor is RDS-internal; client-side
configuration cannot reduce it further.

## Headline result 3: V7 DNS warmup REGRESSED in reboot scenarios

The most interesting (and reportable) finding:

| Config | Reboot business-interruption (5 rounds) |
|---|---|
| v4-current | 0 ms (every round) |
| v5-experimental | 0 ms (every round) |
| v6-aggressive | 0 ms (every round) |
| **v7-dns-warmup** | **5-6 seconds (every round)** |

The application-layer WRITE_FAIL count was zero across all configs (wrapper
4.0.0 catches it), but v7's STATS-gap consistently showed 5-6 seconds of zero
write_ok during reboot, while the other configs maintained throughput.

**Hypothesis**: The DNS warmup thread aggressively re-resolves the cluster
endpoint every second. During a reboot, the writer DNS briefly resolves
to nothing or to a stale IP. The warmup thread caches this momentarily-bad
state in the JVM resolver, which then takes ~5 seconds to invalidate.
Without the warmup thread, the wrapper's internal endpoint cache is
shorter-lived and recovers faster.

**Recommendation**: drop v7 DNS warmup. It hurts more than it helps. The
default Aurora wrapper bg plugin already handles DNS staleness internally;
adding a parallel DNS thread interferes with that mechanism rather than
helping it.

## Headline result 4: V6 aggressive timeout has occasional bad outliers

V6 (`connectTimeout=500ms`) showed 5065 ms WRITE_FAIL in 1 of 5 failover
rounds. The other 4 rounds were 0-2 seconds. This means: connectTimeout=500ms
is on the edge of triggering false-positive failures in a multi-second
RDS internal swap. **Recommendation**: stick with v4's `connectTimeout=1000ms`
for production use; v6 is too aggressive.

## Recommendations distilled

1. **Default**: stay on `v4-current` — it's the proven good config
2. **Avoid v6** — connectTimeout=500ms is too tight, occasional 5s outliers
3. **Avoid v7** — DNS warmup thread regresses reboot scenarios by 5-6 seconds
4. **For new customers on Aurora 3.10+**: even customer-baseline-style
   config may already give zero-downtime BG switchover; verify before
   recommending changes.
